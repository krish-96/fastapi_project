# Alembic Cheat Sheet

A quick reference for managing database migrations with Alembic.

---

# Check Current Database Revision

```bash
alembic current
```

Shows the migration currently applied to the connected database.

---

# View Migration History

```bash
alembic history
```

Displays all migrations in chronological order.

---

# Show Latest Revision (Head)

```bash
alembic heads
```

Displays the latest migration revision available.

---

# Apply All Pending Migrations

```bash
alembic upgrade head
```

Migrates the database to the latest version.

---

# Upgrade One Revision

```bash
alembic upgrade +1
```

Applies only the next pending migration.

---

# Roll Back One Revision

```bash
alembic downgrade -1
```

Reverts the last applied migration.

---

# Roll Back to a Specific Revision

```bash
alembic downgrade <revision_id>
```

Example:

```bash
alembic downgrade 4a1d7d9b3d21
```

---

# Create a New Migration (Auto Generate)

After modifying your SQLAlchemy models:

```bash
alembic revision --autogenerate -m "Add users table"
```

**Always review the generated migration before applying it.**

---

# Create an Empty Migration

```bash
alembic revision -m "Seed default roles"
```

Useful for:

- Data migrations
- Manual SQL
- Stored procedures
- Seed data

---

# Apply Newly Created Migration

```bash
alembic upgrade head
```

---

# Show Migration Details

```bash
alembic show <revision_id>
```

Example:

```bash
alembic show 4a1d7d9b3d21
```

---

# Mark Database Without Running Migrations

If the schema already exists and only the Alembic version table is missing:

```bash
alembic stamp head
```

Stamp a specific revision:

```bash
alembic stamp <revision_id>
```

---

# Show Detailed Migration History

```bash
alembic history --verbose
```

---

# Typical Development Workflow

## 1. Modify SQLAlchemy Models

Update your model definitions.

## 2. Generate Migration

```bash
alembic revision --autogenerate -m "Add email column"
```

## 3. Review Generated Migration

Check the generated file inside:

```
alembic/
└── versions/
```

Verify:

- New columns
- Deleted columns
- Constraints
- Indexes
- Foreign keys

## 4. Apply Migration

```bash
alembic upgrade head
```

## 5. Verify Current Revision

```bash
alembic current
```

# Best Practices

## ✅ Always review autogenerated migrations

Alembic cannot correctly detect operations like:

- Renaming tables
- Renaming columns
- Complex constraint changes

Review the generated Python migration before running it.

---

## ✅ Commit migrations with model changes

Example Git commit:

```
git add app/models/
git add alembic/versions/
git commit -m "Add User profile table"
```

Never commit model changes without the corresponding migration.

---

## ✅ Never edit an already applied migration

Once teammates or production have executed a migration:

❌ Do **not** modify that file.

Instead:

```bash
alembic revision --autogenerate -m "Fix previous migration"
```

Create a new migration.

---

## ✅ Run migrations during deployment

Every deployment should include:

```bash
alembic upgrade head
```

This keeps the database schema synchronized with the application code.

# Downgrade All Migrations

## Downgrade to the Base Revision

```bash
alembic downgrade base
```

or equivalently:

```bash
alembic downgrade 0
```

This reverts **all applied migrations** in reverse order until the database reaches the base (no Alembic-managed
schema).

---

## Check Current Revision

Before downgrading:

```bash
alembic current
```

After downgrading:

```bash
alembic current
```

It should show no current revision (or `None`).

---

## Re-apply All Migrations

To migrate back to the latest version:

```bash
alembic upgrade head
```

---

## Common Downgrade Commands

| Command                           | Description                   |
|-----------------------------------|-------------------------------|
| `alembic downgrade -1`            | Revert the last migration     |
| `alembic downgrade base`          | Revert all migrations         |
| `alembic downgrade 0`             | Same as `base`                |
| `alembic downgrade <revision_id>` | Revert to a specific revision |

---

## ⚠️ Warning

Running:

```bash
alembic downgrade base
```

will execute the `downgrade()` function of **every migration** in reverse order.

If your migrations drop tables or columns, **all corresponding data may be permanently lost**.

Always back up your database before downgrading in production.

# Alembic Offline vs Online Mode

## Online Mode

1. Alembic connects directly to the database using the configured database URL.
2. Migration commands like `alembic upgrade head` execute SQL changes immediately.
3. The database schema is updated automatically by Alembic.
4. Used commonly during development and automated deployments.
5. `context.is_offline_mode()` returns `False` and runs `run_migrations_online()`.

```bash
# Apply migrations directly to the database
alembic upgrade head

# Check current migration version
alembic current

# Create a new migration
alembic revision --autogenerate -m "add users table"
```

---

## Offline Mode

1. Alembic does not connect to the database.
2. It generates SQL migration scripts instead of executing them.
3. Run using commands like `alembic upgrade head --sql`.
4. Useful for production environments where SQL needs review before execution.
5. `context.is_offline_mode()` returns `True` and runs `run_migrations_offline()`.

```bash
# Generate SQL script without applying changes
alembic upgrade head --sql

# Save generated SQL into a file
alembic upgrade head --sql > migration.sql

# Execute generated SQL manually
mysql -u username -p database_name < migration.sql
```
